# Feature: detect a session running a stale claude-mux (launched-version nudge)

## Goal

When the claude-mux script on disk is upgraded but a running session was launched
by an **older** version, that session keeps running the older injection prompt and
launch wrapper (baked in at creation) until it is restarted. Detect this and inject
a one-shot "this session was started by an older claude-mux; restart it to pick up
the new version" notice into the conversation - mirroring the existing Claude Code
upgrade nudge.

Notify-only. No auto-restart (restarting drops Remote Control and interrupts the
user; the decision stays theirs).

**Scope (revised 2026-06-24).** The bare-home incident below showed a second, more
severe failure mode in the same family: a wrapper so stale it relaunches the session
**bare** (no flags, no RC, no injection) rather than merely "older." The feature
therefore carries **two complementary notify-only detectors**: (1) the launched-version
nudge (stale but functional), and (2) a degraded-launch health check (broken/bare
launch). See the Correction and Sibling-design sections.

## Motivating incident: bare home (2026-06-24)

The home session disappeared from Remote Control. Diagnosis found home running a
`claude` process with **none of its flags**: no `--remote-control`, no `--name home`,
no `--append-system-prompt-file` - a bare `claude`. It still showed as `running` in
`claude-mux -l`, and `claude-mux -s` / RC could not reach it.

Chain of events:

1. Home had been running for a long time on a **stale launch wrapper** generated by an
   old claude-mux version. Its wrapper script (and prompt) lived in `$TMPDIR`
   (`claude-launch-fiNCRe`), the pre-fix location from before the prompt moved to
   `<project>/.claudemux-prompt`. macOS reaps `$TMPDIR`.
2. The argentina-property session ran `claude-mux --restart home --fresh` (in the
   background). For a running non-caller session this is
   `shutdown_single_session home` -> `create_claude_session home … fresh`
   (`src/90-dispatch.sh:181-190`). The command printed its normal
   "Restarting 1 session(s)…" success line and exited 0.
3. The result was a **bare** home: the old wrapper's relaunch logic (running with a
   reaped `$TMPDIR` prompt path) brought `claude` back with no flags. Success was
   reported; the session was broken.
4. Recovery was a plain `claude-mux --restart home` from a healthy session, which did
   a full shutdown+create and replaced the wrapper with the **current** one. A
   subsequent `--restart home --fresh` then worked correctly - confirming the **stale
   wrapper was the enabler**, not `--fresh` itself.

Why this matters for this feature: it is the same class of problem (a long-lived
session carrying an old launcher), but a **worse failure mode** than stale injection -
the session comes back not just "older" but **non-functional and invisible to RC**,
with no signal anywhere. Home is the worst case because it is almost never cleanly
recreated (the LaunchAgent only checks "is home alive?", and in-place loops reuse the
same wrapper), so it can carry one wrapper across weeks and many upgrades.

## Research finding that set the mechanism (verified 2026-06-17)

The idea began as "stamp the launching version into the marker file"
(`.claudemux-running-2.0.7`). Verifying the assumptions against the code reversed
the recommendation:

1. **`--update` already restarts.** `do_update` (`src/30-helpers.sh`) ends with
   `exec "$install_path" --restart`, so the in-tool upgrade path re-launches every
   session and re-bakes injection. The stale-injection window therefore exists
   **only for out-of-band upgrades**: `brew upgrade claude-mux`, a manual file
   replace, or editing the script without restarting. (CLAUDE.md "Script, not
   binary" documents exactly this window.)
2. **A relaunch re-bakes injection automatically.** Both launch wrappers regenerate
   the system prompt on every in-place relaunch via
   `--print-system-prompt ... > "$_prompt.new"` (`src/55-session-launch.sh:192`,
   `src/70-start-launch.sh:199`), and the auto-restore tick relaunches dead sessions
   through `launch_single_session`, which calls `build_system_prompt` fresh. So a
   **dead → restored** session always comes back on the *current* injection.

Consequence: the only session that ever runs stale injection is a **live** one.
A dead session's launching version is irrelevant (restore re-bakes). That removes
the one advantage the marker-filename had over a tmux option ("it survives the
process"), because surviving the process is not needed here. → use a tmux user
option, exactly like `@claude-mux-claude-id`.

### Correction (2026-06-24): "a relaunch re-bakes injection" is only true for a *new-enough* wrapper

Finding #2 is the load-bearing assumption for choosing the tmux option over the
marker filename, and the bare-home incident shows it is **only conditionally true**:

- The `--print-system-prompt` re-bake on relaunch is code that lives **inside the
  wrapper**. A wrapper old enough to predate that code does **not** re-bake - it
  relaunches with its own old logic. Home's stale wrapper relaunched a **bare**
  `claude` (no prompt, no flags) precisely because its launch logic predated both the
  re-bake and the `$TMPDIR`->`<project>` prompt move. So "a relaunch always comes back
  on current injection" holds for sessions on a reasonably current wrapper, **not**
  for one on a genuinely ancient wrapper - which is the exact population this feature
  targets.
- This does **not** revive the versioned-marker proposal (its rejection reasons -
  migration cost, mixed semantics - still stand, and the option is still the right
  carrier). What it changes is **scope**: the version comparison alone is
  insufficient, for two reasons:
  1. **Pre-feature blind spot.** `@claude-mux-launched-version` is only set by
     `create_claude_session`/`launch_single_session`. A session on an ancient wrapper
     has the option **unset**, so `detect_claudemux_upgrade` returns a silent no-op
     (see Edge cases). It therefore would **not** have caught bare home - the worst
     case is invisible to the version nudge.
  2. **Wrong signal for the worst failure.** A bare relaunch is not "an older
     version"; it is a process **missing its flags entirely**. No version string can
     express that. It needs a direct health check (next section), not a version
     compare.

So this feature now carries **two complementary detectors**: the version nudge (stale
but functional, below) and a degraded-launch health check (broken/bare launch).

## Design (recommended): a `@claude-mux-launched-version` tmux option

Direct mirror of the Claude Code upgrade detector (`detect_claude_upgrade`,
`src/75-tip-notices.sh:81`), which already stores the launch-time `claude` binary
id in `@claude-mux-claude-id` and compares it each prompt.

**1. Stamp at launch.** Wherever the session options are set
(`@claude-mux-dir` / `@claude-mux-claude-id`), also set
`@claude-mux-launched-version` to the launching script's `$VERSION`. Both the
fresh-launch and the live-session backfill branches set it:
- `src/55-session-launch.sh:73-74` (backfill) and `:92-93` (fresh)
- `src/70-start-launch.sh:95-96` (backfill) and `:234-235` (fresh)

**2. Detect on each prompt.** Add `detect_claudemux_upgrade()` next to
`detect_claude_upgrade()`, called from `on_prompt` in the same place (after the
`Ready?` handshake no-op, before the cheap tip/update guard so it is always-on):

```sh
detect_claudemux_upgrade() {
    local _sess _v0
    _sess=$("$TMUX_BIN" display-message -p '#S' 2>/dev/null) || return 0
    [[ -z "$_sess" ]] && return 0
    _v0=$("$TMUX_BIN" show-options -t "$_sess" -v @claude-mux-launched-version 2>/dev/null)
    [[ -z "$_v0" ]] && return 0                 # pre-feature session: silent no-op
    if version_gt "$VERSION" "$_v0"; then        # on-disk script is newer than the launcher
        echo "[claude-mux — tell the user, in their conversation language]: this session was started by claude-mux $_v0 but $VERSION is now installed; say \"restart this session\" to load the new version."
        "$TMUX_BIN" set-option -t "$_sess" @claude-mux-launched-version "$VERSION" 2>/dev/null
    fi
}
```

- `version_gt` already exists (`src/30-helpers.sh:22`); use the existing comparator.
- One-shot per change: ack by overwriting the option to `$VERSION`, same as the
  Claude Code detector. The user is nudged once per upgrade; if they ignore it, the
  session keeps running stale (their choice) and is not nagged every prompt.
- Always-on, independent of `TIP_OF_DAY` / `UPDATE_CHECK` (it is correctness, not a
  tip). Wire it exactly like `_bin_notice`: compute after the handshake check, and
  prepend to `_out` so it surfaces even when tips/updates are off.

**3. on_prompt wiring.** Compute alongside `_bin_notice`
(`src/75-tip-notices.sh:141`), and include it in every flush path (both-off guard
at :146, no-sid path at :154, and the final prepend at :227). Cleanest: fold both
notices into one "always-on notices" string so the three flush sites stay in sync.

## Sibling design: degraded-launch health check (the bare-home case)

The version nudge above catches "older but functional." The bare-home incident is a
different failure - a session relaunched **missing its flags** - and needs a direct
check on the running process, not a version compare. This is the higher-severity half
of the feature.

**What "healthy" means.** A claude-mux-launched session runs `claude` with, at
minimum, `--remote-control`, `--name <session>`, and
`--append-system-prompt-file <path>`. A bare relaunch has none. So the health signal
is: *does the live `claude` process for this session carry its expected flags?*

**Two candidate signals (pick at build):**

1. **Process-args check.** Inspect the session's `claude` process command line for the
   three flags. Pro: directly observes the real failure (it is exactly how the incident
   was diagnosed - `ps … | grep -- "--name <session>"`). Con: needs to locate the
   session's `claude` pid (walk from the tmux pane pid), and `ps` parsing is
   platform-shaped (fine on macOS, the only supported platform today).
2. **Self-report via a tmux option.** Have the wrapper set
   `@claude-mux-launch-ok=1` (or stamp the flags it actually passed) *only on a
   correctly-flagged launch*. Detection then reads the option, like the version nudge.
   Pro: cheap, no `ps`, same pattern as the rest. **Con: a stale wrapper cannot set an
   option it does not know about** - so a bare relaunch from an ancient wrapper leaves
   the option unset, which is indistinguishable from a pre-feature session. Same blind
   spot as the version nudge. **This makes option (1) the only signal that actually
   sees the worst case.**

**Recommendation: the process-args check.** It is the only detector that observes a
bare launch regardless of which (possibly ancient) wrapper produced it - the version
option and any self-report option are both blind to a wrapper too old to set them.
Wire it into `on_prompt` next to `detect_claude_upgrade` / `detect_claudemux_upgrade`:
if the session's live `claude` lacks `--remote-control` or `--append-system-prompt-file`,
inject a one-shot "this session is running in a degraded state (no Remote Control /
injection); say \"restart this session\" to recover" notice. Notify-only, same as the
rest.

**Caveats to resolve in design:**
- **RC reachability vs flag presence.** "Missing `--remote-control`" is the precise
  cause; "RC not connected" is the user-visible symptom but can have other causes
  (daemon down). Detect on the *flag*, not on a connection probe, to avoid false
  positives.
- **Locating the pid.** The pane's foreground command can be the wrapper `bash`, not
  `claude` (observed in the incident). The check must walk pane pid → descendants to
  find the `claude` process, and tolerate "no claude yet" (mid-relaunch) by no-op.
- **Self-targeting.** The hook runs inside the session it is checking; reading its own
  process args is fine and avoids cross-session `ps` scanning.
- **Should detection auto-trigger a clean restart?** Out of scope - stays notify-only
  per the feature's standing rule (a degraded session is still the user's to restart).
  But note the recovery here is specifically a **full** `--restart` (kill+recreate to
  replace the wrapper), not an in-place relaunch (which would re-run the same broken
  wrapper). The notice should say "restart" plainly; the dispatch already does
  kill+recreate for a non-caller named restart.

**Deeper fix (separate, but worth recording).** Detection nudges after the fact;
the root cause is that `create_claude_session`/`launch_single_session` can *report
success while producing a bare process*. A stronger guarantee is a **post-launch
self-verify**: after launching, confirm the new `claude` carries its flags (or that
the wrapper is the current one) and fail loudly / retry if not, so a bare launch never
silently reports success. That is a correctness fix in the launch path, complementary
to (and arguably more important than) the notify-only detector. Capture as a candidate;
decide scope with the architect.

## Considered alternative (rejected): version in the marker filename

`.claudemux-running-<version>` instead of the bare `.claudemux-running`. Rejected:

- **Migration cost across every marker reader.** Today readers use a fixed-name
  existence test: `should_be_alive` (`src/50:169`), `autorestore_status`
  (`src/50:182`), and the wrapper teardown `rm -f "$_marker"` (`src/55`/`src/70`).
  All would have to become globs (`compgen -G "$dir/.claudemux-running-*"`), and the
  writer would need a "remove any existing `.claudemux-running-*` then touch the new
  name" step to preserve the single-marker invariant. More surface, more sharp edges.
- **The benefit isn't load-bearing.** The only thing the filename buys over a tmux
  option is a launching-version that outlives the process - and per the research
  finding above, a dead session's launching version is irrelevant (restore re-bakes
  injection). So the cost buys nothing the live-session use case needs.
- **Mixes semantics.** The marker is a pure presence flag whose meaning is its
  location (folder = which session). Encoding data in its name erodes that and the
  marker-file philosophy in CLAUDE.md ("boolean flags: empty `touch`-ed file").

If a *post-death* launching version is ever genuinely needed (e.g. crash diagnostics
"it died under 2.0.7"), the right home is the per-session restore-state JSON
(`$RESTORE_STATE_DIR/<name>.json`), which already carries structured runtime state -
not the marker filename.

## Edge cases / risks

| Case | Handling |
|---|---|
| Session launched before this feature (no option set) | `show-options -v` returns empty → silent no-op (same as `detect_claude_upgrade` for a pre-feature session). |
| `Ready?` handshake turn | The handshake no-op (`on_prompt` exits at `_is_handshake == 1`, `src/75:136`) runs first, so the nudge never fires on a handshake and never burns its one-shot ack. MUST stay ordered after that check. |
| In-tool `--update` | Auto-restarts (`exec ... --restart`), which re-stamps the option to the new `$VERSION` at relaunch → notice never fires for the in-tool path. Correct: there is nothing stale to nudge. |
| Downgrade (on-disk older than launcher) | `version_gt "$VERSION" "$_v0"` is false → no notice. Intended. |
| Not in tmux / option unset | `display-message`/`show-options` fail → `return 0`, no output. |
| Home session | Gets the option like any session; restart nudge applies (home is restartable). No marker is involved, so no special-casing needed. |
| Degraded launch on an ancient wrapper (bare home) | Version nudge MISSES it (option unset → silent no-op). The degraded-launch process-args check is the detector that catches it. |
| Mid-relaunch (no `claude` child yet) | Process-args check finds no `claude` under the pane → no-op (do not false-flag a session that is still coming up). |
| Pane foreground is the wrapper `bash`, not `claude` | Walk pane pid → descendants to find the `claude` pid (observed in the incident); do not read the pane's foreground command directly. |

## Out of scope

- Auto-restart on detection (notify-only by design - applies to both detectors).
- The launch-path **self-verify** (fail-loudly-on-bare-launch) is noted under the
  degraded-launch sibling as a complementary correctness fix, but is a **separate**
  decision for the architect; this feature ships the notify-only detectors.
- Changing the `.claudemux-running` marker in any way (it stays a bare presence flag).
- A `-l`/`-L` column showing each session's launched version (possible follow-up;
  the option makes it cheap, but it is display polish, not the nudge).
- Detecting wrapper/injection changes specifically (we use the coarse `$VERSION`;
  a version bump is the proxy for "something a restart would pick up," which is the
  same proxy the Claude Code detector uses with the binary id).
