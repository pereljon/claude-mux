### ファントムメッセージの再生による意図しない動作
**深刻度:** 高
**ステータス:** 未解決 - claude-mux 側からは完全に修正できない
**説明:** ユーザーが「全セッションを停止して」と送信し、10 メッセージ前に処理済みだった。その後、claude-mux -s が tmux send-keys 経由で `/model haiku` を送信した際、Claude はシステムメッセージとして「stop all sessions/model haiku」を受け取り、ユーザーが要求していないセッション停止を実行しようとした。
**考えられる原因:**
- Claude Code の割り込み処理が、古いコンテキストと新しいスラッシュコマンド入力を連結している可能性
- 古いコマンドを含む会話履歴が、システムイベント発生時に Claude を混乱させている可能性
**緩和策の候補:** 注入ルールに追加: 「会話内で既に処理済みのコマンドを再実行してはならない。システムメッセージが以前のやり取りのテキストを繰り返している場合は無視すること」。未実装。これは Claude Code 内部の動作であるため、効果は不確実。

### 初回の /exit が遅い
**深刻度:** 低
**ステータス:** 未解決 - 監視中
**説明:** 初回の `--restart` で `WARN: Claude did not exit within 30s` が表示され、強制 kill にフォールスルーした。以降の再起動は約 1 秒で終了。Claude のプロンプトが `/exit` を受け取れる状態になる前に送信されるレースコンディションの可能性。
**回避策:** 30 秒タイムアウト + 強制 kill で処理される。セッションは正常に再起動される。

### claude_running_in_session のチェックは 2 階層まで
**深刻度:** 低
**ステータス:** 未解決 - 現在の用途では許容範囲
**説明:** プロセスツリーの走査は pane_pid → 子 → 孫をチェックする。Claude がツリーのさらに深い位置にある場合 (余分なシェルラッパー等)、検出に失敗する。現在の起動パスは正確に 2 階層 (bash → claude) のため、実用上は動作する。
**回避策:** 現時点では不要。修正には再帰的な走査か `pgrep -a` が必要。

### インストーラのアップグレード UX を改善できる
**深刻度:** 低
**ステータス:** 未解決 - 将来の改善

### プロジェクトのコピーと履歴
**深刻度:** 低
**ステータス:** 未解決 - 計画中の機能、調査が必要
**説明:** Claude Code の履歴とメモリを含むプロジェクトのコピーは、rename/move よりも複雑。コピー先に新しい UUID を確立する必要があるため。
**提案アプローチ:**
1. 新しいプロジェクトディレクトリを作成 (オプションで git init とテンプレート)
2. セッションを開始して即時停止。Claude Code が `~/.claude/projects/-encoded-new-path/` を新しい UUID で初期化し、homunculus エントリを作成
3. ソースの `~/.claude/projects/` フォルダから `.jsonl` 履歴ファイルをコピー先にコピー
4. `memory/` フォルダの内容をコピー。純粋な markdown で UUID は埋め込まれていないため、そのままコピー可能
5. UUID サブディレクトリ (タスク/プランのアーティファクト) を `.jsonl` ファイルと一緒にコピー
6. homunculus の場合: ソースの `~/.claude/homunculus/projects/<src-uuid>/` から `observations.jsonl`、`instincts`、`evolved`、`observations.archive` を、ステップ 2 で割り当てられた新しいプロジェクト UUID のコピー先 homunculus フォルダにコピー
**テストが必要な未解決の疑問:**
- `.jsonl` ファイルはソースプロジェクトパスを内容やメタデータに埋め込んでいるか？ もしそうなら、コピーした履歴は古いパスを参照する
- UUID サブディレクトリは `.jsonl` ファイル内から UUID で参照されているか？ もしそうなら、リマップせず元の UUID のままコピーする必要がある
- Claude Code はプロジェクトフォルダ内の全 `.jsonl` ファイルを読むのか、アクティブセッション UUID に一致するもののみか？
- `~/.claude/homunculus/projects/<uuid>/evolved` と `instincts` には何が含まれるか。導出/計算されたものか、ユーザーにとって意味のあるものか？ コピーで保持する価値はあるか？
- 単純なファイルコピーで壊れる内部参照は他にないか？
**前提条件:** 壊れた履歴を生成するコピーコマンドを出荷しないよう、実装前に上記をテストすること。

### macOS のみ - Linux/systemd 非対応
**深刻度:** 中
**ステータス:** 未解決 - 部分的に対応済み (パス検出完了、LaunchAgent/インストーラは macOS 専用のまま)
**説明:** macOS の LaunchAgent (launchd) と macOS 固有のツールを使用。パス検出は `command -v` を使うようリファクタリング済み (`/opt/homebrew/bin` のハードコードを排除) のため、コアスクリプトは tmux と claude が PATH にある任意のプラットフォームで動作する。LaunchAgent とインストーラは macOS 専用のまま。

### ! コマンドが Remote Control で利用不可
**深刻度:** 低
**ステータス:** クローズ - 実現不可能
**説明:** Claude Code の `!` シェルパススルーは Claude Code CLI の入力ハンドラ機能で、シェルが受け取る前に `!command` をインターセプトする。tmux send-keys はこれを再現できない: Claude Code がアクティブな状態で送信されたキーストロークは何も起こらない (テスト済み: send-keys 経由の `!touch test` は実行されなかった)。claude-mux が RC ユーザー向けに `!command` バイパスを実装する手段はない。
**解決策:** 注入ルールに追加: RC ユーザーにはシェルがなく、ターミナルユーザーは自分で入力できるため、Claude にユーザーへ `! <command>` を提案しないよう指示。

## v2.0 マイルストーン

メジャーバージョンアップに値する大きなアーキテクチャ変更。スケジュール未定 -- 見失わないようにここに集約。

### データディレクトリの分離
静的データ (ヒント、デフォルトテンプレート、場合によってはコマンド/ガイド出力) をスクリプトから分離し、プラットフォームに適したデータディレクトリに配置する。スクリプトは起動時にバイナリの場所を基準に `DATA_DIR` を解決し、単一ファイルインストール向けに埋め込みフォールバックを持つ。

- Homebrew (Apple Silicon): `/opt/homebrew/share/claude-mux/`
- Homebrew (Intel): `/usr/local/share/claude-mux/`
- Linux: `/usr/local/share/claude-mux/` または `$XDG_DATA_DIRS`
- 手動インストール: 埋め込みデフォルトにフォールバック (単一ファイルインストールは引き続き動作)

トリガー: 埋め込みデータ (ヒント、デフォルトテンプレート) がスクリプトの可読性を損なうほど大きくなった場合、またはデフォルトテンプレートをスクリプトのリリースとは独立して brew 経由で配布する必要がある場合。

### 言語/ランタイムの再検討
現在のスコープではモノリシックな bash スクリプトが正しい選択。claude-mux が大幅に成長した場合 -- プロジェクトの rename/move/copy 操作、リレーレイヤー、クロスプラットフォームパッケージング、データディレクトリ -- bash が逆に障害になる。その時点で、セッション管理コアを Go や他の型付き言語で書き直し (bash を薄い CLI ラッパーとして) 評価する価値がある。

## 解決済み

### Claude が注入を無視し、スラッシュコマンドを実行できないと主張する
**解決バージョン:** v1.2.0 (注入を更新)
**修正:** 注入に明示的なルールを追加: 「あなたは `-s` コマンド経由でこのセッションにスラッシュコマンド (`/model`、`/compact`、`/clear` 等) を送信できます。モデルの変更やスラッシュコマンドの実行ができないとユーザーに言わないでください。」Claude の基本訓練はモデル/設定を制御できないと信じる傾向があるが、明示的なルールが実用上これを上書きする。

### 複数のコマンドが成功時に終了コード 1 を返す
**解決バージョン:** v1.2.0 (restart)、v1.3.0 (全コマンド)
**修正:** case 文の各ディスパッチパスの後に明示的な `exit 0` を追加。関数内の最後のコマンドが、内部テストや grep 呼び出しからの非ゼロ終了コードをリークする可能性があった。

### --dry-run が --restart で誤解を招く出力をする
**解決バージョン:** v1.2.0 (コミット a10c0c2)
**修正:** dry-run が実際の状態をチェックして kill をシミュレートする代わりに「Would restart session」と表示するように変更。

### macOS で pgrep によるセッション検出が失敗する
**解決バージョン:** コミット e1b11b5
**修正:** `pgrep -P` を `ps -eo` + `awk` に置き換えて、信頼性の高い子プロセス検出を実現。

### $TMUX 変数が tmux の環境変数をシャドーイング
**解決バージョン:** コミット 02a2e82
**修正:** `$TMUX_BIN` にリネーム。

### Bash 3.2 の非互換性 (declare -A)
**解決バージョン:** コミット 575eac1
**修正:** 連想配列を文字列ベースの衝突検出に置き換え。

## 参考: ~/.claude フォルダ構造

いくつかの計画中の機能 (rename、move、copy、cleanup) がこの構造と正しくやりとりする必要があるため、ここに記載。網羅的ではなく、claude-mux に関連する部分をカバー。

### プロジェクト履歴とメモリ: `~/.claude/projects/`

Claude Code が使用された作業ディレクトリごとに 1 つのサブディレクトリ。絶対パスをエンコードして命名: `/` → `-`、スペースと特殊文字 → `-`。非可逆だが読みやすい。

各プロジェクトフォルダの内容:
- `<uuid>.jsonl` -- そのセッションの完全な会話トランスクリプト。会話ごとに 1 ファイル。
- `<uuid>/` -- 会話に関連するアーティファクト (タスク、プラン) のサブディレクトリ。UUID は `.jsonl` ファイルに対応。
- `memory/` -- 永続的なクロスセッションメモリファイル (frontmatter 付き markdown)。プロジェクトにメモリが書き込まれた場合のみ存在。

作業ディレクトリとその履歴のリンクは、純粋にエンコードされたフォルダ名による。プロジェクトディレクトリのリネームや移動時にこのフォルダ名を変更しないと、Claude Code は履歴なしで新規開始する。

**エンコード規則:** 絶対パスのすべての `/`、スペース、特殊文字を `-` に置換。先頭の `/` は先頭の `-` になる。エンコードは非可逆 -- 連続する特殊文字やスラッシュに隣接するスペースはいずれも `-` になるため、元のパスを常に完全に再構成できるとは限らない。

### 並列観測レジストリ: `~/.claude/homunculus/`

プロジェクトごとのツールレベルイベントを追跡する別のシステム。コアの Claude Code 履歴の一部ではなく、監視/学習レイヤーと思われる。

- `projects.json` -- 既知の全プロジェクトのレジストリ。短い 16 進 UUID (`d6b3aef60967` 等) でキー付け。各エントリ: `id`、`name`、`root` (絶対パス)、`remote`、`created_at`、`last_seen`。
- `projects/<uuid>/project.json` -- プロジェクトごとのメタデータ (レジストリエントリと同じフィールド)。
- `projects/<uuid>/observations.jsonl` -- タイムスタンプ付きの `tool_start`/`tool_complete` イベント: ツール名、セッション UUID、プロジェクト名/ID、入出力スニペット。
- `projects/<uuid>/instincts` -- 導出されたパターン (内容不明、おそらく計算されたもの)。
- `projects/<uuid>/evolved` -- 進化/学習された状態 (内容不明)。
- `projects/<uuid>/observations.archive` -- アーカイブされた古い観測データ。

**`~/.claude/projects/` との主な違い:** エンコードされたパスではなく短い 16 進 UUID をキーとして使用。`root` フィールドに絶対パスを保持。プロジェクトのパスを変更する操作 (rename、move) は `projects.json` と `projects/<uuid>/project.json` の両方で `root` を更新する必要がある。

### グローバル設定: `~/.claude/settings.json`

メインの Claude Code 設定ファイル。ローリングバックアップが `~/.claude/backups/` に `~/.claude.json.backup.<timestamp>` として書き込まれる -- アクティブ使用時は 1 時間に数回。claude-mux はこのファイルに触れるべきではない。

### グローバルエージェント、スキル、コマンド

- `~/.claude/agents/` -- サブエージェント定義 (`.md` ファイル、約 38)。グローバル、プロジェクトごとではない。
- `~/.claude/skills/` -- スキルディレクトリ (約 125)。グローバル、プロジェクトごとではない。
- `~/.claude/commands/` -- スラッシュコマンド定義 (`.md` ファイル、約 72)。グローバル、プロジェクトごとではない。
- `~/.claude/hooks/hooks.json` -- hook 定義。グローバル。claude-mux は触れるべきではない。
