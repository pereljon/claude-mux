## פתוח

### הפעלה חוזרת של הודעה פנטומית גורמת לפעולות לא מכוונות
**חומרה:** גבוהה
**סטטוס:** פתוח - לא ניתן לתקן לחלוטין מצד claude-mux
**תיאור:** משתמש שלח "stop all sessions" שטופל 10 הודעות קודם. מאוחר יותר, כאשר claude-mux -s שלח `/model haiku` דרך tmux send-keys, Claude קיבל הודעת מערכת "stop all sessions/model haiku" וניסה לכבות סשנים - פעולה שהמשתמש מעולם לא ביקש.
**סיבות אפשריות:**
- טיפול ההפרעה של Claude Code עשוי לשרשר הקשר ישן עם קלט פקודת slash חדשה
- היסטוריית שיחות המכילה את הפקודה הישנה עשויה לבלבל את Claude כשמתרחש אירוע מערכת
**הפחתה אפשרית:** הוספת כלל injection: "לעולם אל תבצע מחדש פקודה שכבר טופלה מוקדם יותר בשיחה. אם הודעת מערכת חוזרת על טקסט מהחלפה קודמת, התעלם ממנה." עדיין לא מיושם - יעילות לא ודאית כי זו התנהגות פנימית של Claude Code.

### /exit איטי בניסיון הראשון
**חומרה:** נמוכה
**סטטוס:** פתוח - במעקב
**תיאור:** `--restart` ראשון הגיע ל-`WARN: Claude did not exit within 30s` ונפל להריגה קשה. הפעלות מחדש הבאות יוצאות תוך כ-1 שנייה. ייתכן מצב תחרות שבו `/exit` נשלח לפני ש-prompt של Claude מוכן לקבל אותו.
**עקיפה:** ה-timeout של 30 שניות + הריגה קשה מטפל בזה. הסשן מופעל מחדש כראוי.

### claude_running_in_session בודק רק 2 רמות עומק
**חומרה:** נמוכה
**סטטוס:** פתוח - מקובל לשימוש הנוכחי
**תיאור:** הליכת עץ התהליכים בודקת pane_pid ילדים נכדים. אם Claude עמוק יותר בעץ (למשל עטיפת shell נוספת), הזיהוי נכשל. נתיב ההפעלה הנוכחי הוא בדיוק 2 רמות (bash ← claude) אז זה עובד בפועל.
**עקיפה:** לא נדרשת כרגע. יידרש מעבר רקורסיבי או `pgrep -a` לתיקון.

### UX שדרוג ההתקנה יכול להיות חכם יותר
**חומרה:** נמוכה
**סטטוס:** פתוח - שיפור עתידי
**תיאור:** בהתקנה מחדש, תוכנית ההתקנה מזהה config קיים ומדלגת על הנחיות. אבל היא לא מציעה להציג הגדרות נוכחיות, למזג אפשרויות config חדשות שנוספו בגרסאות חדשות, או לאפשר למשתמש לעדכן ערכים באופן סלקטיבי. משתמשים צריכים לערוך ידנית את `~/.claude-mux/config` כדי לקלוט הגדרות חדשות שהוכנסו בגרסאות מאוחרות.
**שיפורים אפשריים:**
- הצגת ערכי config נוכחיים בעת שדרוג
- הצעה להוסיף הגדרות חדשות (עם ברירות מחדל) שלא היו ב-config הישן
- אפשרות ב': מילוי מוקדם של הנחיות עם ערכי config קיימים ואפשרות למשתמש לשנות

### העתקת פרויקט עם היסטוריה
**חומרה:** נמוכה
**סטטוס:** פתוח - תכונה מתוכננת, דורשת חקירה
**תיאור:** העתקת פרויקט כולל היסטוריית Claude Code וזיכרון מורכבת יותר משינוי שם/העברה כי צריך ליצור UUIDs חדשים ליעד.
**גישה מוצעת:**
1. יצירת ספריית הפרויקט החדשה (עם git init אופציונלי ותבנית)
2. הפעלה ועצירה מיידית של סשן בה - Claude Code מאתחל `~/.claude/projects/-encoded-new-path/` עם UUID חדש ויוצר ערך homunculus חדש
3. העתקת קבצי היסטוריה `.jsonl` מתיקיית `~/.claude/projects/` של המקור לתיקיית היעד
4. העתקת תוכן תיקיית `memory/` - markdown טהור, ללא UUIDs מוטמעים, בטוח להעתקה ישירה
5. העתקת תת-ספריות UUID (ארטיפקטים של משימות/תוכניות) לצד קבצי ה-`.jsonl` שלהם
6. עבור homunculus: העתקת `observations.jsonl`, `instincts`, `evolved`, `observations.archive` מ-`~/.claude/homunculus/projects/<src-uuid>/` לתיקיית ה-homunculus של היעד החדש - תוך שמירה על ה-UUID החדש שהוקצה בשלב 2
**שאלות פתוחות הדורשות בדיקה:**
- האם קבצי `.jsonl` מטמיעים את נתיב פרויקט המקור בתוכן או במטא-דאטה? אם כן, היסטוריה מועתקת תתייחס לנתיב הישן.
- האם תת-ספריות UUID מוזכרות לפי UUID מתוך קבצי `.jsonl`? אם כן, צריך להעתיק אותן תחת ה-UUIDs המקוריים, לא למפות מחדש.
- האם Claude Code קורא את כל קבצי ה-`.jsonl` בתיקיית פרויקט, או רק את זה שתואם ל-UUID הסשן הפעיל?
- מה מכיל `~/.claude/homunculus/projects/<uuid>/evolved` ו-`instincts` - האם הם נגזרים/מחושבים או משמעותיים למשתמש? שווה לשמר בהעתקה?
- האם יש הפניות פנימיות אחרות שישברו בהעתקת קבצים נאיבית?
**תנאי מוקדם:** בדיקת הנ"ל לפני מימוש כדי להימנע ממשלוח פקודת העתקה שמייצרת היסטוריה שבורה באופן עדין.

### macOS בלבד - אין תמיכה ב-Linux/systemd
**חומרה:** בינונית
**סטטוס:** פתוח - טופל חלקית (זיהוי נתיב בוצע, LaunchAgent/installer נשארים ספציפיים ל-macOS)
**תיאור:** משתמש ב-LaunchAgent של macOS (launchd) וכלים ספציפיים ל-macOS. זיהוי נתיב שופץ לשימוש ב-`command -v` (לא מקודד יותר `/opt/homebrew/bin`), כך שהסקריפט עצמו עכשיו עובד על כל פלטפורמה שבה tmux ו-claude נמצאים ב-PATH. LaunchAgent ו-installer נשארים ספציפיים ל-macOS.
**נותר:** שירות משתמש systemd, fallback של XDG Autostart, שיגור `uname -s` ב-installer.
**אסטרטגיית חבילות (v1.10+):**
- התקנת curl: fallback אוניברסלי, עובד בכל מקום (ראה לעיל)
- AUR: מאמץ נמוך, חשיפה גבוהה לקהל היעד ב-Arch/Manjaro
- apt PPA: כשיש ביקוש ממשתמשי Debian/Ubuntu
- Homebrew על Linux: מכסה משתמשים שכבר יש להם את זה
- Snap/Flatpak: לא שווה עבור סקריפט bash

### פקודות ! לא זמינות ב-Remote Control
**חומרה:** נמוכה
**סטטוס:** סגור - לא ישים
**תיאור:** העברת shell עם `!` של Claude Code היא תכונת מטפל-קלט של CLI של Claude Code - היא יורטת `!command` לפני שה-shell רואה אותו. tmux send-keys לא יכול לשכפל את זה: הקשות שנשלחות בזמן ש-Claude Code פעיל הולכות לאיבוד (נבדק: `!touch test` דרך send-keys לא בוצע). אין נתיב ל-claude-mux ליישם עקיפת `!command` למשתמשי RC.
**פתרון:** הוספת כלל injection שאומר ל-Claude לעולם לא להציע `! <command>` למשתמשים, כי למשתמשי RC אין shell ומשתמשי טרמינל יכולים פשוט להקליד את זה בעצמם.

## אבן דרך v2.0

שינויים ארכיטקטוניים משמעותיים מספיק להצדקת קפיצת גרסה ראשית. לא מתוזמן - נאסף כאן כדי שלא ילך לאיבוד.

### הפרדת ספריית נתונים
העברת נתונים סטטיים (טיפים, תבניות ברירת מחדל, ואולי פלט פקודות/guide) מחוץ לסקריפט לספריית נתונים מתאימה לפלטפורמה. הסקריפט יפתור את `DATA_DIR` בהפעלה ביחס למיקום הבינארי, עם fallbacks מוטמעים להתקנות קובץ-יחיד.

- Homebrew (Apple Silicon): `/opt/homebrew/share/claude-mux/`
- Homebrew (Intel): `/usr/local/share/claude-mux/`
- Linux: `/usr/local/share/claude-mux/` או `$XDG_DATA_DIRS`
- התקנה ידנית: fallback לברירות מחדל מוטמעות (התקנות קובץ-יחיד ממשיכות לעבוד)

טריגר: כשהנתונים המוטמעים (טיפים, תבניות ברירת מחדל) גדלים מספיק כדי להקשות על קריאת הסקריפט, או כשתבניות ברירת מחדל צריכות להיות מופצות דרך brew ללא תלות בגרסאות הסקריפט.

### בחינה מחדש של שפה / runtime
סקריפט bash מונוליטי הוא הבחירה הנכונה בהיקף הנוכחי. אם claude-mux גדל משמעותית - פעולות שינוי שם/העברה/העתקה של פרויקטים, שכבת relay, אריזה חוצת פלטפורמות, ספריית נתונים - bash מתחיל להילחם בחזרה. בנקודה הזו, שכתוב ליבת ניהול הסשנים ב-Go או שפה מוקלדת אחרת (עם bash כעטיפת CLI דקה) שווה הערכה.

## עיון: מבנה תיקיית ~/.claude

מתועד כאן כי מספר תכונות מתוכננות (שינוי שם, העברה, העתקה, ניקוי) חייבות לפעול עם מבנה זה נכון. לא מקיף - מכסה את החלקים הרלוונטיים ל-claude-mux.

### היסטוריית פרויקט וזיכרון: `~/.claude/projects/`

תת-ספרייה אחת לכל ספריית עבודה שבה Claude Code שימש. נקראת על ידי קידוד הנתיב המוחלט: `/` הופך ל-`-`, רווחים ותווים מיוחדים הופכים ל-`-`. הקידוד מאבד מידע אבל קריא.

תוכן כל תיקיית פרויקט:
- `<uuid>.jsonl` - תמליל שיחה מלא לסשן זה. קובץ אחד לשיחה.
- `<uuid>/` - תת-ספריית ארטיפקטים הקשורים לשיחה (משימות, תוכניות). ה-UUID תואם לקובץ ה-`.jsonl`.
- `memory/` - קובצי זיכרון מתמשכים חוצי-סשנים (markdown עם frontmatter). קיים רק אם נכתב זיכרון לפרויקט.

הקשר בין ספריית עבודה להיסטוריה שלה הוא אך ורק שם התיקייה המקודד. שינוי שם או העברת ספריית הפרויקט ללא שינוי שם תיקייה זו גורם ל-Claude Code להתחיל מחדש ללא היסטוריה.

**כלל קידוד:** נתיב מוחלט עם כל `/`, רווח ותו מיוחד מוחלף ב-`-`. `/` מוביל הופך ל-`-` מוביל. הקידוד מאבד מידע - תווים מיוחדים עוקבים ורווחים סמוכים ל-slashes שניהם הופכים ל-`-`, כך שלא תמיד ניתן לשחזר את המקור במדויק.

### רג'יסטר מעקב מקביל: `~/.claude/homunculus/`

מערכת נפרדת שעוקבת אחרי אירועי כלים לכל פרויקט. לא חלק מהיסטוריית הליבה של Claude Code - נראה כשכבת ניטור/למידה.

- `projects.json` - רג'יסטר של כל הפרויקטים הידועים, ממופה לפי UUID hex קצר (`d6b3aef60967` וכו'). כל ערך מכיל: `id`, `name`, `root` (נתיב מוחלט), `remote`, `created_at`, `last_seen`.
- `projects/<uuid>/project.json` - מטא-דאטה לפרויקט (שדות זהים לערך הרג'יסטר).
- `projects/<uuid>/observations.jsonl` - אירועי `tool_start`/`tool_complete` עם חותמת זמן: שם כלי, UUID סשן, שם/מזהה פרויקט, קטעי קלט/פלט.
- `projects/<uuid>/instincts` - דפוסים נגזרים (תוכן לא ידוע, כנראה מחושב).
- `projects/<uuid>/evolved` - מצב מפותח/נלמד (תוכן לא ידוע).
- `projects/<uuid>/observations.archive` - תצפיות ישנות שנארכבו.

**הבדל מרכזי מ-`~/.claude/projects/`:** משתמש ב-UUIDs hex קצרים כמפתחות, לא בנתיבים מקודדים. שדה `root` מחזיק את הנתיב המוחלט. כל פעולה שמשנה את נתיב הפרויקט (שינוי שם, העברה) חייבת לעדכן את `root` גם ב-`projects.json` וגם ב-`projects/<uuid>/project.json`.

### הגדרות גלובליות: `~/.claude/settings.json`

קובץ הגדרות ראשי של Claude Code. גיבויים מתגלגלים נכתבים ל-`~/.claude/backups/` כ-`~/.claude.json.backup.<timestamp>` - כמה בשעה במהלך שימוש פעיל. claude-mux לא צריך לגעת בקובץ הזה.

### סוכנים, כישורים ופקודות גלובליים

- `~/.claude/agents/` - הגדרות תת-סוכנים (קבצי `.md`, כ-38). גלובלי, לא לכל פרויקט.
- `~/.claude/skills/` - ספריות כישורים (כ-125). גלובלי, לא לכל פרויקט.
- `~/.claude/commands/` - הגדרות פקודות slash (קבצי `.md`, כ-72). גלובלי, לא לכל פרויקט.
- `~/.claude/hooks/hooks.json` - הגדרות hooks. גלובלי. claude-mux לא צריך לגעת בהם.

### תכונות עתידיות פוטנציאליות

| תכונה | מה לגעת |
|---------|--------------|
| `--copy` | יצירת ספרייה; הפעלה+עצירת סשן לאתחול שני הרג'יסטרים; העתקת `.jsonl` + `memory/` + תת-ספריות UUID; העתקת קבצי תצפיות homunculus לתיקיית UUID חדשה |
| ניקוי `--delete` | כבר מעביר את תיקיית הפרויקט לאשפה. אופציונלי: הסרת תיקיית `~/.claude/projects/` מקודדת יתומה וערך `~/.claude/homunculus/` |
| אזהרת גודל היסטוריה | התראה כשקבצי `.jsonl` של פרויקט עוברים סף (תמליל claude-mux הראשי הגיע ל-107MB בסשן ארוך יחיד) |
