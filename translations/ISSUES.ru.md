## Открытые

### Фантомное воспроизведение сообщений вызывает непреднамеренные действия
**Серьёзность:** Высокая
**Статус:** Открыто; невозможно полностью исправить на стороне claude-mux
**Описание:** Пользователь отправил "stop all sessions", которое было обработано 10 сообщений назад. Позже, когда claude-mux -s отправил `/model haiku` через tmux send-keys, Claude получил системное сообщение "stop all sessions/model haiku" и попытался остановить сессии, чего пользователь не запрашивал.
**Возможные причины:**
- Обработка прерываний в Claude Code может конкатенировать старый контекст с вводом новой слэш-команды
- История разговора, содержащая старую команду, может путать Claude при возникновении системного события
**Возможное решение:** Добавить правило в инъекцию: "Никогда не выполняй повторно команду, уже обработанную ранее в разговоре. Если системное сообщение повторяет текст из предыдущего обмена, игнорируй его." Пока не реализовано; эффективность неясна, так как это внутреннее поведение Claude Code.

### Медленный /exit при первой попытке
**Серьёзность:** Низкая
**Статус:** Открыто; мониторинг
**Описание:** Первый `--restart` вызвал `WARN: Claude did not exit within 30s` и перешёл к принудительному завершению. Последующие перезапуски завершаются за ~1 с. Возможно, это состояние гонки, когда `/exit` отправляется до того, как промпт Claude готов к приёму.
**Обходной путь:** Тайм-аут 30 с + принудительное завершение решает проблему. Сессия перезапускается корректно.

### claude_running_in_session проверяет только 2 уровня глубины
**Серьёзность:** Низкая
**Статус:** Открыто; приемлемо для текущего использования
**Описание:** Обход дерева процессов проверяет pane_pid -> дочерние -> внучатые. Если Claude находится глубже в дереве (например, дополнительная обёртка шелла), обнаружение не работает. Текущий путь запуска ровно 2 уровня (bash -> claude), поэтому на практике всё работает.
**Обходной путь:** Не требуется. Для исправления потребуется рекурсивный обход или `pgrep -a`.

### UX установщика при обновлении можно улучшить
**Серьёзность:** Низкая
**Статус:** Открыто; будущее улучшение
**Описание:** При повторной установке установщик обнаруживает существующую конфигурацию и пропускает вопросы. Но он не предлагает показать текущие настройки, смерджить новые параметры конфигурации, добавленные в более новых версиях, или позволить пользователю выборочно обновить значения. Пользователи должны вручную редактировать `~/.claude-mux/config`, чтобы подключить новые настройки, появившиеся в следующих версиях.
**Возможные улучшения:**
- Показывать текущие значения конфигурации при обновлении
- Предлагать добавить новые настройки (со значениями по умолчанию), которых не было в старой конфигурации
- Вариант Б: предзаполнять вопросы существующими значениями конфигурации и позволять пользователю их менять

### Копирование проекта с историей
**Серьёзность:** Низкая
**Статус:** Открыто; запланированная функция, требует исследования
**Описание:** Копирование проекта вместе с историей и памятью Claude Code сложнее, чем переименование/перемещение, потому что для целевого проекта нужно создать новые UUID.
**Предлагаемый подход:**
1. Создать каталог нового проекта (с опциональным git init и шаблоном)
2. Запустить и сразу остановить сессию в нём: Claude Code инициализирует `~/.claude/projects/-encoded-new-path/` с новым UUID и создаёт новую запись в homunculus
3. Скопировать файлы истории `.jsonl` из папки `~/.claude/projects/` источника в целевую папку
4. Скопировать содержимое папки `memory/`: чистый markdown, без встроенных UUID, безопасно для прямого копирования
5. Скопировать подкаталоги UUID (артефакты задач/планов) вместе с их файлами `.jsonl`
6. Для homunculus: скопировать `observations.jsonl`, `instincts`, `evolved`, `observations.archive` из `~/.claude/homunculus/projects/<src-uuid>/` в папку homunculus нового проекта, сохраняя новый UUID проекта, назначенный на шаге 2
**Открытые вопросы, требующие тестирования:**
- Содержат ли файлы `.jsonl` путь исходного проекта в содержимом или метаданных? Если да, скопированная история будет ссылаться на старый путь.
- Ссылаются ли подкаталоги UUID по UUID из файлов `.jsonl`? Если да, их нужно копировать под оригинальными UUID, а не перемаппить.
- Читает ли Claude Code все файлы `.jsonl` в папке проекта, или только тот, что соответствует UUID активной сессии?
- Что содержат `~/.claude/homunculus/projects/<uuid>/evolved` и `instincts`: производные/вычисленные данные или пользовательски значимые? Стоит ли сохранять при копировании?
- Есть ли другие внутренние ссылки, которые сломаются при наивном копировании файлов?
**Предварительное условие:** Протестировать вышеизложенное перед реализацией, чтобы не выпустить команду копирования, которая создаёт незаметно повреждённую историю.

### Временная метка ответа
**Серьёзность:** Низкая
**Статус:** Открыто; обсудить перед реализацией
**Описание:** Необязательная переменная конфигурации (`REPLY_TIMESTAMP=false` по умолчанию), которая внедряет инструкцию в системный промпт, указывая Claude начинать каждый ответ с текущей даты и времени через `date '+%Y-%m-%d %H:%M'`.
**Компромисс:** Требует вызова bash-инструмента в начале каждого ответа (небольшие накладные расходы). Альтернатива: внедрить время старта сессии в промпт (бесплатно, но дрейфует в длинных сессиях).
**Примечание:** Инструкция в CLAUDE.md конкретного проекта (как в аналитическом шаблоне) это более лёгкая версия: только в проектах, которым это нужно. Переменная конфигурации делает это глобальным.

### Подача в homebrew-core для листинга на brew.sh

**Статус:** Будущее; ожидание роста аудитории
**Описание:** claude-mux в настоящее время распространяется через персональный tap (`pereljon/tap`). Чтобы появиться на brew.sh, нужно быть принятым в homebrew-core. Порог известности Homebrew обычно требует несколько сотен звёзд на GitHub, прежде чем заявка на утилиту-шелл-скрипт будет принята; заявки с малым числом звёзд быстро закрываются.
**Когда будет готово:**

### Поддержка установки через curl (macOS + Linux)
**Серьёзность:** Низкая
**Статус:** Решено в v1.10.0: реализована установка через curl, добавлен workflow для релизных артефактов, README обновлён

### Только macOS; нет поддержки Linux/systemd
**Серьёзность:** Средняя
**Статус:** Открыто; частично решено (определение путей переработано, LaunchAgent/установщик остаются специфичными для macOS)
**Описание:** Использует LaunchAgent macOS (launchd) и специфические для macOS инструменты. Определение путей переработано для использования `command -v` (больше не хардкодит `/opt/homebrew/bin`), поэтому ядро скрипта теперь работает на любой платформе, где tmux и claude находятся в PATH. LaunchAgent и установщик остаются специфичными для macOS.
**Остаётся:** systemd user unit, фолбэк XDG Autostart, диспетчеризация `uname -s` в установщике.
**Стратегия пакетирования (v1.10+):**
- Установка через curl: универсальный фолбэк, работает везде (см. выше)
- AUR: малые усилия, большой охват целевой аудитории на Arch/Manjaro
- apt PPA: когда появится спрос от пользователей Debian/Ubuntu
- Homebrew на Linux: для пользователей, которые уже его используют
- Snap/Flatpak: не стоит усилий для bash-скрипта

### Команды ! недоступны в Remote Control
**Серьёзность:** Низкая
**Статус:** Закрыто; не осуществимо
**Описание:** Пробрось шелла `!` в Claude Code это функция обработчика ввода CLI Claude Code: она перехватывает `!command` до того, как шелл его увидит. tmux send-keys не может это повторить: нажатия клавиш, отправленные когда Claude Code активен, никуда не идут (тест: `!touch test` через send-keys не выполнилось). Нет пути для claude-mux реализовать обход `!command` для пользователей RC.
**Решение:** Добавлено правило в инъекцию, указывающее Claude никогда не предлагать `! <command>` пользователям, так как у пользователей RC нет шелла, а пользователи терминала могут просто набрать это сами.

## Milestone v2.0

Архитектурные изменения, достаточно значительные для мажорного обновления версии. Не запланированы; собраны здесь, чтобы не потерялись.

### Разделение каталога данных
Переместить статические данные (советы, шаблоны по умолчанию, возможно вывод команд/справки) из скрипта в платформо-подходящий каталог данных. Скрипт будет определять `DATA_DIR` при запуске относительно расположения бинарного файла, с встроенными фолбэками для однофайловых установок.

- Homebrew (Apple Silicon): `/opt/homebrew/share/claude-mux/`
- Homebrew (Intel): `/usr/local/share/claude-mux/`
- Linux: `/usr/local/share/claude-mux/` или `$XDG_DATA_DIRS`
- Ручная установка: фолбэк на встроенные значения по умолчанию (однофайловые установки продолжают работать)

Триггер: когда встроенные данные (советы, шаблоны по умолчанию) станут достаточно большими, чтобы затруднить чтение скрипта, или когда шаблоны по умолчанию нужно будет поставлять через brew независимо от релизов скрипта.

### Переоценка языка/среды выполнения
Монолитный bash-скрипт это правильный выбор при текущем масштабе. Если claude-mux значительно вырастет: операции переименования/перемещения/копирования проектов, релейный слой, кросс-платформенная упаковка, каталог данных — bash начинает сопротивляться. В этот момент стоит оценить переписывание ядра управления сессиями на Go или другом типизированном языке (с bash в качестве тонкой CLI-обёртки).

## Решено

### Claude игнорирует инъекцию и утверждает, что не может выполнять слэш-команды
**Решено в:** v1.2.0 (инъекция обновлена)
**Исправление:** Добавлено явное правило в инъекцию: "You CAN send slash commands (`/model`, `/compact`, `/clear`, etc.) to this session via the `-s` command. Never tell the user you cannot change models or run slash commands." Базовое обучение Claude склоняет его считать, что он не может управлять своей моделью/настройками; явное правило переопределяет это на практике.

### Несколько команд возвращают код выхода 1 при успехе
**Решено в:** v1.2.0 (restart), v1.3.0 (все команды)
**Исправление:** Добавлен явный `exit 0` после каждого пути диспетчеризации в операторе case. Последняя команда в функции может протечь ненулевым кодом выхода из внутренних тестов или вызовов grep.

### --dry-run даёт вводящий в заблуждение вывод для --restart
**Решено в:** v1.2.0 (коммит a10c0c2)
**Исправление:** Dry-run теперь показывает "Would restart session" вместо симуляции kill с последующей проверкой реального состояния.

### Обнаружение сессий не работает с pgrep на macOS
**Решено в:** коммит e1b11b5
**Исправление:** Заменён `pgrep -P` на `ps -eo` + `awk` для надёжного обнаружения дочерних процессов.

### Переменная $TMUX затеняла переменную окружения tmux
**Решено в:** коммит 02a2e82
**Исправление:** Переименована в `$TMUX_BIN`.

### Несовместимость с Bash 3.2 (declare -A)
**Решено в:** коммит 575eac1
**Исправление:** Ассоциативные массивы заменены на обнаружение коллизий на основе строк.

## Справка: структура папки ~/.claude

Документировано здесь, потому что несколько запланированных функций (переименование, перемещение, копирование, очистка) должны корректно взаимодействовать с этой структурой. Не исчерпывающе: охватывает только части, относящиеся к claude-mux.

### История и память проекта: `~/.claude/projects/`

Один подкаталог на каждый рабочий каталог, в котором использовался Claude Code. Именуется путём кодирования абсолютного пути: `/` -> `-`, пробелы и специальные символы -> `-`. С потерями, но читаемо.

Содержимое каждой папки проекта:
- `<uuid>.jsonl` — полная расшифровка разговора для этой сессии. Один файл на разговор.
- `<uuid>/` — подкаталог артефактов, связанных с разговором (задачи, планы). UUID совпадает с файлом `.jsonl`.
- `memory/` — постоянные файлы памяти между сессиями (markdown с метаданными). Присутствуют только если память была записана для проекта.

Связь между рабочим каталогом и его историей основана исключительно на закодированном имени папки. Переименование или перемещение каталога проекта без переименования этой папки заставляет Claude Code начать с чистого листа без истории.

**Правило кодирования:** абсолютный путь, в котором каждый `/`, пробел и специальный символ заменяется на `-`. Ведущий `/` становится ведущим `-`. Кодирование с потерями: последовательные специальные символы и пробелы рядом со слэшами становятся `-`, поэтому оригинал не всегда может быть точно восстановлен.

### Параллельный реестр наблюдаемости: `~/.claude/homunculus/`

Отдельная система, отслеживающая события на уровне инструментов по каждому проекту. Не часть основной истории Claude Code: по-видимому, это слой мониторинга/обучения.

- `projects.json` — реестр всех известных проектов, индексированных коротким hex UUID (`d6b3aef60967` и т.д.). Каждая запись содержит: `id`, `name`, `root` (абсолютный путь), `remote`, `created_at`, `last_seen`.
- `projects/<uuid>/project.json` — метаданные проекта (те же поля, что и в записи реестра).
- `projects/<uuid>/observations.jsonl` — события `tool_start`/`tool_complete` с временными метками: имя инструмента, UUID сессии, имя/id проекта, фрагменты ввода/вывода.
- `projects/<uuid>/instincts` — производные шаблоны (содержимое неизвестно, вероятно вычислено).
- `projects/<uuid>/evolved` — эволюционировавшее/выученное состояние (содержимое неизвестно).
- `projects/<uuid>/observations.archive` — архивированные старые наблюдения.

**Ключевое отличие от `~/.claude/projects/`:** Использует короткие hex UUID как ключи, а не закодированные пути. Поле `root` содержит абсолютный путь. Любая операция, изменяющая путь проекта (переименование, перемещение), должна обновить `root` как в `projects.json`, так и в `projects/<uuid>/project.json`.

### Глобальная конфигурация: `~/.claude/settings.json`

Основной файл настроек Claude Code. Ротационные бэкапы записываются в `~/.claude/backups/` как `~/.claude.json.backup.<timestamp>`: несколько в час при активном использовании. claude-mux не должен трогать этот файл.

### Глобальные агенты, навыки, команды

- `~/.claude/agents/` — определения подагентов (файлы `.md`, ~38). Глобальные, не по проектам.
- `~/.claude/skills/` — каталоги навыков (~125). Глобальные, не по проектам.
- `~/.claude/commands/` — определения слэш-команд (файлы `.md`, ~72). Глобальные, не по проектам.
- `~/.claude/hooks/hooks.json` — определения хуков. Глобальные. claude-mux не должен их трогать.

### Потенциальные будущие функции

| Функция | Что нужно затронуть |
|---------|---------------------|
| `--copy` | Создать каталог; запустить и сразу остановить сессию для инициализации обоих реестров; скопировать `.jsonl` + `memory/` + подкаталоги UUID; скопировать файлы наблюдений homunculus в новую папку UUID |
| Очистка `--delete` | Уже перемещает папку проекта в корзину. Опционально: удалить осиротевшую закодированную папку `~/.claude/projects/` и запись `~/.claude/homunculus/` |
| Предупреждение о размере истории | Оповещение, когда файлы `.jsonl` проекта превышают порог (основная транскрипция claude-mux достигла 107 МБ в одной длинной сессии) |
